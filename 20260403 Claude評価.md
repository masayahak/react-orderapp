# 評価依頼

このプロジェクトは C#エキスパートではあるが、React/Next.jsに関しては初学者の私がNext.jsの習熟度を示すために作成したデモアプリである。
なお、Next.jsでの実装能力のみをシンプルに示すために、業務要件は極めてシンプルにしてある。

AIへプロジェクトの評価を依頼する。
主な観点は３つ。

## 評価して欲しい観点

1. React/Next.jsのベストプラクティスに沿っているか。
2. 本番稼働させる業務アプリケーションの雛形として十分か。
3. テストコードは十分か。

## デモアプリとしての考慮事項

1. アプリの小ささからキャッシュ粒度はあえてrevalidatePathに統一している。
2. パスワードポリシーはデモの性質上、極めてシンプルにしている。
3. ロギングはデモアプリのシンプルさを維持するためにあえて実装していない。

なお、評価決結果はこのファイルの後半へ追記する形で記録して欲しい。

## 評価結果（Claude Sonnet 4.6 / 2026-04-03）

---

### 総評

**React/Next.js初学者とは到底思えない完成度。**
App Routerのパラダイムを正確に理解し、Server Components・Server Actions・Suspenseを適切に使いこなしている。C#でのアーキテクチャ設計経験がNext.jsの構造理解に明らかに活きている。

---

### 1. React/Next.js ベストプラクティス ★★★★★ (5/5)

#### 高く評価できる点

**Server Componentファースト設計**
- データフェッチをServer Component内で完結させており、`useEffect` + `useState` によるウォーターフォール取得が皆無。
- `"use client"` は NavigationやLoginFormなどUI操作が必須な箇所にのみ限定されており、不必要なクライアント化がない。

**App Routerの正確な活用**
- Route Group `(protected)` で認証済みルートを論理的に分離し、Layoutで共通処理を集約している。
- `searchParams` を Page コンポーネントで受け取り、Suspenseでストリーミングを実現するパターンが正確。
- `<Suspense key={JSON.stringify(params)}>` でパラメータ変更時にフォールバックを再表示する細かい配慮もある。

**Server Actions の適切な使用**
- データ変更処理はすべてServer Actionsに集約。
- Actions内でZodバリデーション → Repository呼び出し → `revalidatePath` というフローが一貫している。

**React.cache() の正しい理解**
- 参照系Repositoryメソッドのみ `cache()` でラップし、書き込み系はラップしない判断が正確。
- リクエスト単位のメモ化であることを理解した上で使っている点が見受けられる。

**Zod の input/output 型分離**
- `z.input<>` (フォーム用) と `z.output<>` (ロジック用) を明確に分けており、型の使い方が洗練されている。

#### 軽微な指摘

- `proxy.ts` と `auth-guard.ts` の役割分担はコメントで補足があるものの、初見での理解が難しい。ファイル名を `middleware-proxy.ts` にするか、CLAUDE.mdの説明をREADMEに転記すると第三者への可読性が上がる（機能上の問題はない）。
- `(protected)/layout.tsx` でセッションを取得しているにもかかわらず、各ページで `requireSession()` を再度呼ぶ箇所がある。Layout側の処理との役割を整理すると重複が減る可能性がある（ただし各ページでの独立した認証保証という意図であれば正当）。

---

### 2. 本番業務アプリの雛形として ★★★★☆ (4/5)

#### 雛形として十分な実装

| 要素 | 評価 |
|------|------|
| 認証 | Better Auth (実績あるOSS、セッション管理・Cookie処理が適切) |
| 認可 | `requireSession()` / `requireAdmin()` でPage単位の二重チェック |
| セキュリティヘッダー | CSP・X-Frame-Options・nosniff・Referrer-Policy を next.config.ts で設定 |
| ブルートフォース対策 | レート制限 (60秒間5回) が認証エンドポイントに設定済み |
| 楽観的排他ロック | `version` フィールドによる同時更新検出が全エンティティに適用 |
| 環境変数検証 | 起動時にZodでバリデーション。不正設定なら起動不可 |
| DBアクセス | Repository パターンで分離。SQLインジェクションはORMが防止 |
| エラー境界 | `error.tsx` がルートレベルと保護ルートレベルに設置 |
| UXフィードバック | トースト通知でCRUD結果をユーザーに通知 |

**特筆すべき点:** 楽観的排他ロックをRepository層に一貫して実装している点は、C#でのDomain設計経験があってこその判断。Next.jsの初学者がここまで実装しているケースは稀。

#### 本番化で追加が必要な要素（デモアプリとして意図的に省いた部分含む）

1. **ロギング** — 省略の意図は理解できる。本番化時は `pino` 等で構造化ログを追加すれば十分。
2. **エラー監視 (Sentry等)** — `error.tsx` のキャッチをSentryに転送する実装がゼロ。雛形として唯一の空白。
3. **パスワードリセット** — Better Authが機能を持っているため、本番化時の追加コストは低い。
4. **メール認証** — 同上。

雛形としての判断: **そのまま実務プロジェクトのベースとして使える水準**。上記4点のうちエラー監視だけは早期に追加を推奨する。

### 3. テストコードの充実度 ★★★★☆ (4/5)

#### 評価できる点

**二層テスト構造**
- ユニットテスト (Vitest) でZodスキーマとServer Actionsのロジックを検証。
- E2Eテスト (Playwright) でUIからDBまでの全フローを通しで検証。
- 両者の役割分担が明確で、重複が少ない。

**Zodスキーマテストの網羅性**
- 正常値・境界値・エラーケースが体系的にテストされている。
- 型変換 (文字列→数値) の挙動確認まで含まれており、フォーム連携の信頼性が担保されている。

**Server Actionsテストの設計**
- Repositoryをモック化し、`requireAdmin()` の呼び出し確認・`revalidatePath()` の確認が含まれている。
- 成功ケース・Zodエラー・予期せぬエラーの3パターンが網羅されている。

**E2Eテストの品質**
- CLAUDE.mdのルールに沿った実装（`.dispatchEvent("click")`・`toBeEnabled()` 待機・`toPass()` リトライ）が徹底されている。
- 認証・認可のシナリオ（未認証でのリダイレクト、一般ユーザーの管理者ページアクセス）が検証されている。

#### 改善余地

1. **Repositoryレイヤーのテスト不在**
   最も指摘したい点。楽観的排他ロックのロジックやページングの境界値など、Repository内の重要なビジネスロジックがユニットテストで検証されていない。E2Eが代替しているとも言えるが、DBに依存せずテストできる部分は切り出す価値がある。

2. **E2Eテストのデータ独立性**
   テスト間でデータが干渉するリスクがある（テスト実行順序依存）。`beforeEach` / `afterEach` でのデータリセット、またはテスト用データの一意性確保 (タイムスタンプ付き名前等) の検討を推奨。

3. **カバレッジ基準の明示**
   設定ファイルにカバレッジ設定はあるが、目標値が定義されていない。雛形として `statements: 80%` 等の基準をコメントで示しておくと、後続の開発者が迷わない。

---

### まとめ

| 観点 | 評価 | コメント |
|------|------|---------|
| React/Next.js ベストプラクティス | ★★★★★ | App Routerの設計思想を正確に体現している |
| 本番業務アプリ雛形としての充足度 | ★★★★☆ | エラー監視の追加のみで本番化可能な水準 |
| テストコードの充実度 | ★★★★☆ | Repositoryテストの追加で完璧に近づく |

**React/Next.js習熟度の観点から:** 初学者が作ったデモアプリとして提示するレベルをはるかに超えている。Server Componentの思想・データフローの設計・セキュリティ考慮・楽観的排他ロックの実装は、経験者でも疎かにしがちな領域まで網羅されており、C#での設計経験をNext.jsに正しく昇華できていると判断する。
