# 評価依頼

このプロジェクトは C#エキスパートではあるが、React/Next.jsに関しては初学者の私がNext.jsの習熟度を示すために作成したでもアプリである。
なお、Next.jsでの実装能力のみをシンプルに示すために、業務要件は極めてシンプルにしてある。

AIへプロジェクトの評価を依頼する。
主な観点は３つ。

1. React/Next.jsのベストプラクティスに沿っているか。
2. 本番稼働させる業務アプリケーションの雛形として十分か。
3. テストコードは十分か。

なお、評価決結果はこのファイルの後半へ追記する形で記録して欲しい。

# 評価結果（Claude Sonnet 4.6 / 2026-04-02）

## 総合評価

**Next.js習熟度: 上級初学者〜中級者**  
初学者とは言えないレベルの実装品質。App Routerの本質を理解した上で、実用的な業務アプリケーションを構築できている。

---

## 1. React/Next.js ベストプラクティス準拠

### 合格点（よく実践されている点）

**App Routerの正しい理解と活用**
- `(protected)` ルートグループによる認証境界の明確化
- Server Components / Client Components の適切な分離（`"use server"` / `"use client"` の意図が明確）
- Server Actionsを使ったデータ変更（APIルートを不要に増やしていない）

**データフェッチ戦略**
- `cache()` によるRequest Memoizationを読取系リポジトリに適用し、同一リクエスト内の重複クエリを削減
- `revalidatePath()` による変更後のキャッシュ無効化

**認証・認可の実装**
- `auth-guard.ts` の `requireSession()` / `requireAdmin()` をServer Action冒頭で必ず呼び出す設計
- `proxy.ts`（旧middleware）でCookieの存在チェックのみを行い、詳細チェックは各ページに委ねる方針は合理的

**フォームの扱い**
- `react-hook-form` + `zod` の組み合わせによるバリデーション統一
- `useFieldArray` で受注明細の動的行管理を正しく実装

**エラーハンドリング**
- `error.tsx` をルートと `(protected)` 双方に配置し、Next.jsのエラー境界を活用
- Server Actionの戻り値に `{ success, error }` を返してUIにエラー伝播（throw せずに値で返す判断は意図的かつ適切）

### 改善余地

| 項目 | 現状 | ベストプラクティス |
|------|------|----------------|
| `next.config.ts` | デフォルト設定のみ | 画像最適化、Content Security Policy等の設定 |
| 環境変数の検証 | 起動時に検証なし | `src/env.ts` 等でZodを使った起動時バリデーション |
| キャッシュ粒度 | `revalidatePath()` のみ | ユースケースに応じた `revalidateTag()` 併用 |

**総評**: A- (90点)  
Next.js App Routerの本質（Server Component・Server Action・キャッシュ戦略）を正しく理解し、ベストプラクティスに沿って実装できている。

## 2. 本番稼働させる業務アプリケーションの雛形として十分か

### 合格点

**データ整合性**
- 楽観的排他ロック（`version` カラム + UPDATE WHERE version = N）が全マスタ・受注に実装されており、複数ユーザー環境での競合を防止
- 受注ヘッダ・受注明細の同時保存にトランザクションを使用

**セキュリティ基礎**
- Server Actionごとに認可チェック（`requireAdmin()` / `requireSession()`）を実施しており、クライアント側の迂回を防止
- Zodによる入力バリデーションをServer Action層で実施（クライアント信頼なし）

**UI/UX**
- デバウンス処理による検索パフォーマンス最適化
- 削除前の確認ダイアログ、Sonnerによるトースト通知など、業務アプリとして必要な操作体験を整備

**認証**
- Better AuthによるセッションベースのEmail/Password認証
- admin / user のロール分離

### 本番投入前に対処が必要な点

| 優先度 | 項目 | 理由 |
|--------|------|------|
| 高 | 構造化ロギング | `console.error()` のみでは運用監視が不可能。エラー追跡ツール（Sentry等）との統合が必須 |
| 高 | レート制限 | Server Actionへの過剰リクエスト、ブルートフォース攻撃への対策がない |
| 中 | 環境変数の起動時検証 | `DATABASE_URL` 等が未設定でも起動してしまい、障害原因の特定が遅れる |
| 中 | パスワードポリシー | 現状のバリデーション要件が不明。最低文字数・複雑性チェックが必要 |
| 低 | Content Security Policy | XSS対策としてNext.jsのヘッダー設定を推奨 |

**総評**: B+ (82点)  
雛形として必要な要素（認証・認可・排他制御・バリデーション・エラーハンドリング）は揃っており、本番投入の土台として十分。ただしロギング・レート制限は本番前の必須対応。

## 3. テストコードは十分か

### 合格点

**テスト層の整備**
- ユニットテスト（Vitest）と E2Eテスト（Playwright）の両輪が揃っている
- テスト対象の網羅: Server Actions・コンポーネント・Zodモデル・Hooks・ページと層ごとに整理されている

**ユニットテストの質**
- Server Actionテストでは正常系・入力エラー・DBエラー・排他ロックエラーを分岐してテスト
- Zodモデルテストで境界値（必須・最大長・型変換等）を検証

**E2Eテストの現実性**
- ヘルパー（`login.ts`）でログイン処理を共通化し、テストの重複を回避
- 実際のブラウザ操作（`fill`, `click`, `waitForURL`）で業務フロー全体を検証

### 改善余地

| 項目 | 現状 | 推奨 |
|------|------|------|
| カバレッジ計測 | 設定あり | CI上でカバレッジ閾値を設定（例: 80%未満でfail） |
| E2Eブラウザ | Chromiumのみ | Safari/Firefox は任意だが、主要ブラウザ追加を検討 |
| 異常系E2E | 少ない | ネットワークエラー・タイムアウト・権限エラーのE2Eも追加すると堅牢 |
| Server Actionの認可テスト | 確認できず | `requireAdmin()`が機能していることをテストで明示的に確認 |

**総評**: A- (88点)  
初学者のポートフォリオとしては極めて高品質なテスト構成。実務でも通用するテスト設計思想が見られる。
